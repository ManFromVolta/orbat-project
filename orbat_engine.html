Концепция

ORBAT v0.2 — В разработке

ORBAT — игронезависимый фреймворк управления боем двух и более сторон. Логика боя полностью абстрагирована от движка и ассетов, что позволяет подключать её к любой игре через адаптерный слой.

Принцип: одна и та же логика должна описывать и общевойсковой бой дивизии NATO, и битву при Хельмовой пади.

Цепочка

Общая логика → Интерпретация в движке → Подключение ассетов

BattleState (Lua) └── Commander (роль в каждом юните) └── Unit (листовой узел) ├── Order (приказ = намерение) ├── ROE (поведение при выполнении) ├── Perception (что знает юнит) └── Assets (изменяют capability)

Архитектура юнита

Юнит — листовой узел графа командования. Атомарная единица исполнения приказов.

Четыре механических действия

Тип юнита определяется комбинацией флагов, не жёстким enum. Танк = MOVE+SHOOT+SEE. Разведка = MOVE+SEE. САУ = MOVE+BLAST.
ДействиеОписаниеПример
MOVEПеремещается в пространствеПехота, танки, кавалерия
SEEОбнаруживает объекты в радиусеРазведка, наблюдатель
SHOOTАтакует прямым огнём (LOS)Стрелки, танки, лучники
BLASTАтакует зону без LOSАртиллерия, катапульта, авиация

Контракт юнита

Unit { id, side, domain -- ground / air / sea state -- idle / moving / engaging / suppressed / dead hp -- 0.0 – 1.0 position -- {x, y, z} facing -- 0 – 360° actions -- [MOVE, SEE, SHOOT, BLAST] move_speed vision_range attack_range attack_damage engage_targets -- ground / air / sea logistics { supply -- 0.0 – 1.0 supply_rate threshold } parent -- id командира order -- текущий приказ report -- пакет наверх perception { ... } }

Система приказов

Ключевое решение: приказ = намерение. Поведение при выполнении определяет ROE, а не приказ.
ТипПараметрыСмысл
MOVEposition {x,z}Выдвинуться в точку
HOLDposition {x,z}Удерживать точку
SUPPORTtarget: unit_id │ {x,z}Поддержать юнит или зону
Order { L1: type -- MOVE │ HOLD │ SUPPORT L2: target -- unit_id или {x,z} L3: radius -- для SUPPORT по зоне L4: expire -- секунды до протухания │ null }

Что делатьOrder (приказ)
Как себя вестиROE (доктрина, каскадная)

Ассеты

Ассет — пассивный объект который меняет capability юнита.

ТипПоведениеПример
ConsumableУничтожаетсяБоеприпасы, топливо, еда
AttachedСледует за юнитомГрузовик, БТР, лошадь
StaticОстаётся на местеМост, база, укрепление
Asset { id type -- consumable / attached / static state -- available / in_use / destroyed owner, assigned_to capacity payload { speed_bonus, supply_amount, supply_type armor, comms_bonus } on_owner_destroyed: if assigned_to.state = dead → assigned_to = null, state = available }

ИсточникЧто даёт

SEEspotted[] в vision_range, terrain рядом
net_dataReport'ы юнитов, данные Commander'а
РежимУсловиеИсточник решений

Autonomous — инициатива снизу. Именно здесь партизан решает выдвинуться на помощь без приказа.

Система приоритетов

ПриоритетУровеньОписание
1 — ObjectiveВысшийГлавная задача. Не меняется никогда.
2 — OrderСреднийТекущий приказ от Commander'а.
3 — SurvivalУсловныйТолько когда выполнение физически невозможно.
Survival не перебивает Objective. Юнит не убегает при первом выстреле.

World Data и ZoneMap

World Data — объективная

Один, общий, движок. terrain_map, passability, connections. В Teardown динамический — мост снесли, connections обновился.

ZoneMap — субъективная

Своя у каждой стороны. Два противника видят разное. Green / Yellow / Red. Confidence + expires.

Любой contact report → green → yellow Подтверждённый, confidence>0.7 → yellow → red Верификация чистая → red → yellow → green
Зона маршрутаРежимМеханика
GreenPull по расписаниюCommander запрашивает периодически
YellowPush по событиямЮнит докладывает при контакте
RedPush + верификацияМаксимальное внимание
spotted[] всегда перекрывает zone_cache. Вижу сам — доверяю глазам, не карте.

ROE — Правила применения силы

Ключевое решение: stance не является частью приказа. ROE наследуется каскадом от HQ вниз до юнита.
ПараметрДиапазонОписание
risk_tolerance0.0–1.0Готовность действовать при угрозе
loyalty0.0–1.0Реакция на состояние союзников
initiative0.0–1.0Автономия при degraded/offline comms
can_execute = path существует AND supply > threshold AND цель достижима threat_level = threat_raw * (1 - risk_tolerance) social = priority_delta * loyalty search_drive = priority_delta * initiative * (1 - best_asset_score)

Дерево решений

Шаг 0: нет приказа + comms offline initiative > 0.7 AND priority_delta > 0.6 → AUTO_ORDER │ HOLD Шаг 1: can_execute > 0.6 AND threat < 0.7 → EXECUTE Шаг 2: почему не можем? supply < threshold → SEEK_ASSET path blocked → FIND_ALT_PATH capability = 0 → HOLD search_drive > 0.3 → MOVE(search_area) Шаг 3: threat > 0.7 can_execute > 0.5 → ENGAGE │ FALLBACK Шаг 4: social_delta > order.weight → SUPPORT/MOVE Шаг 5: comms offline initiative * can_execute > 0.5 → автономно │ HOLD

Commander

Commander — роль в каждом юните, не отдельная сущность. При гибели — следующий по рангу принимает командование автоматически.

Уничтожение командиров — легитимная тактика. Система деградирует органически.
CommanderRole { is_commander, rank, group_id baseline_threat, roe_overrides } next = max(rank) среди unit.state != dead в group_id

Цикл Commander'а (каждые 30 тиков)

1. aggregate(spotted[]) → update(group_state) 2. combat_power < fallback_threshold → FALLBACK всей группе 3. unit.asset_score < 0.3 → назначить ассет через asset_policy 4. known_enemies изменились → пересчитать приказы 5. send_report(parent)

Искажение

Усталый юнит под огнём докладывает 10 танков. Их было 3.

qualification = base_skill * unit.hp * (1-threat_level) * logistics.supply
Report { L1: sender_id L2: timestamp L3: type -- push_event │ pull_response L4: priority -- 1 критично │ 2 важно │ 3 рутина L5: content L6: confidence L7: expires L8: delivery { method, carrier_id, delivered } }
