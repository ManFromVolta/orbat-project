01 Концепция

ORBAT — игронезависимый фреймворк управления боем двух и более сторон. Логика боя полностью абстрагирована от движка и ассетов.

Принцип: одна и та же логика должна описывать и общевойсковой бой дивизии NATO, и битву при Хельмовой пади.

ORBAT разрабатывается как самостоятельная стратегическая игра-песочница на Godot Engine. Мир игры — Оррим, эпоха 1952 н.э.

Граф командования

BattleState
└── HQ
     └── Commander ← роль в каждом юните
          └── Unit / CompositeUnit ← листовой узел
               ├── Order      приказ = намерение + опц. директива (L5)
               ├── ROE        поведение при выполнении
               ├── Perception что знает юнит
               └── Assets     inventory / container / mutable

03 Архитектура юнита

Юнит — листовой узел графа командования. Атомарная единица исполнения приказов. Техника — это ассеты-контейнеры. Юнитом является экипаж.

Принадлежность к типу — танк, пехота, конница — хранится в profile и используется движком и UI, но не алгоритмами боевой логики.

Четыре базовых действия

MOVE
Перемещение в пространстве. Пехота, танки, кавалерия, корабли.
SEE
Обнаружение объектов в радиусе. Разведка, наблюдатель, БПЛА.
SHOOT
Атака прямым огнём (LOS). Стрелки, танки, лучники.
BLAST
Атака зоны без LOS. Артиллерия, катапульта, миномёты.

Контракт юнита

Unit {
  id, side, domain              # ground / air / sea
  profile { descriptor, era, visual }   # для UI, не для логики
  state                         # idle / moving / engaging / dead
                                # | mounted (в экипаже контейнера)
                                # | in_cargo (в грузовом отсеке)
  hp                            # 0.0 – 1.0
  position                      # Vector3
  location {
    mode:         free | mounted | in_cargo
    container_id: null | asset_id
    role:         null | 'driver' | 'gunner' | ...
  }
  base_abilities { move, see, shoot, blast }  # фиксированы
  modifiers[]                                 # стек, пересчёт перед каждым решением
  move_speed, vision_range, attack_range, attack_damage, engage_targets
  logistics { supply, supply_rate, threshold }
  parent, order, report, perception { ... }
}

04 Система abilities: base + modifiers

abilities (0.0 – 1.0)
Степень доступности действия при текущем состоянии
MOVE = 0.05 → ползком под огнём
Характеристики
Эффективность при выполнении
move_speed, attack_damage, vision_range
abilities — «может ли и насколько свободно». Характеристики — «насколько сильно». Перемножаются при выполнении, хранятся и изменяются независимо.
abilities = clamp(base_abilities + sum(modifiers), 0.0, 1.0)
            # вычисляется перед каждым решением ROE

# Suppressed — модификатор, не состояние:
{ source: 'suppressed', move: -0.50, see: -0.35, shoot: -0.45, blast: 0.0,
  expires: 'suppression_source_gone' }

05 Composite Unit

CompositeUnit образуется автоматически, когда ассет-контейнер укомплектован экипажем выше crew_threshold. Является полноценным юнитом в графе Commander'а. Идентифицируется по контейнеру — смена экипажа или груза не меняет id.

БМП идентифицируется по корпусу машины, а не по составу экипажа.
CompositeUnit extends Unit {
  id = f(container.id)          # идентичность = контейнер
  composite {
    container_id
    crew         [unit_ids]     # заполненные crew_slots
    cargo_units  [unit_ids]     # десант / пассажиры
    cargo_assets [asset_ids]    # груз
  }
  # domain наследуется от контейнера
  # abilities.base = container.base_abilities
  # modifiers += crew_skill_modifiers + cargo_weight_penalty
}

# Образование:
crew_filled / required_slots >= crew_threshold
  → form(composite), commander.group заменяет crew на composite

# Растворение (обратимо):
crew_filled / required_slots < crew_threshold
  → dissolve: crew выходят как отдельные юниты, контейнер = ассет

Жизненный цикл: БМП

СостояниеCommander видитЛогика
Экипаж 3/3 + десант 8БМП-1 [crew:3 cargo:8]composite, domain: ground
Десант высадилсяБМП-1 [crew:3] + Отделение АБМП остаётся composite
Убит водитель (обязательный)dissolve → 2 crew + asset:БМП-1Ниже threshold, composite распался
Экипаж полностью убитasset:БМП-1 [uncrewed]Не юнит — ассет без статуса

Abilities composite unit

composite.abilities:
  base    = container.base_abilities
  # Вклад экипажа
  modifiers += crew_skill_modifiers(crew, container.crew_slots)
  # Некомплект: пропорциональный штраф
  if fill_ratio < 1.0: modifiers += { shoot: -(1-fill_ratio)*0.5, ... }
  # Груз снижает мобильность
  modifiers += { move: -cargo_weight_penalty }

# Десант in_cargo не использует свои abilities,
# если только container.allows_cargo_fire = true

06 Типология ассетов

Ассет — любое материальное средство, расширяющее и/или модифицирующее возможности юнита. Лошадь, танк, автомат, копьё, буксируемая пушка — всё это ассеты. Ассеты классифицируются по двум ортогональным осям.

Ось subtype — структурная роль

SubtypeОпределениеПримеры
inventory Используется конкретным юнитом, меняет его характеристики. Не содержит других юнитов. Личное оружие, броня, боеприпасы, бинокль, паёк
container Принимает юниты и/или другие ассеты. При укомплектовании образует composite unit. Танк, БМП, корабль, орудийная позиция, БПЛА-носитель
mutable Переключается между режимами inventory и container в зависимости от состояния (stowed / deployed). Станковый пулемёт, буксируемая пушка, развёртываемый пункт связи

Ось behavior — поведение

BehaviorОписаниеПримеры
consumableУничтожается после использованияБоеприпасы, топливо, еда
attachedСледует за юнитом / compositeЛошадь, БТР, грузовик
staticОстаётся на местеМост, укрепление, орудие на позиции
subtype и behavior ортогональны. БМП: container + attached. Буксируемая пушка в боевом положении: container + static. Патроны: inventory + consumable.

07 Container layer

container {
  crew_slots [
    { role: 'driver',     required: true  },
    { role: 'gunner',     required: true  },
    { role: 'commander',  required: false }
  ]
  cargo_unit_slots:  int          # десант / пассажиры
  cargo_asset_slots: int          # по объёму / весу
  current_crew:         [unit_ids]
  current_cargo_units:  [unit_ids]
  current_cargo_assets: [asset_ids]
  base_abilities { move, see, shoot, blast }
  crew_threshold:    float        # доля обязат. слотов для composite
  allows_cargo_fire: bool         # может ли десант стрелять изнутри
}

Ассет-контейнер может содержать другие контейнеры в cargo_asset_slots. Десантный корабль перевозит БМП (container) и боеприпасы (inventory). Вложенный контейнер активируется как composite только после выгрузки.

Частичная погрузка/разгрузка не обязательно разрушает composite. Высадка десанта из БМП не выводит её из строя — экипаж управляет машиной. Потеря обязательных crew slots — выводит.

08 Mutable ассеты

Mutable ассет переключается между двумя режимами. В stowed — переносится как inventory с нулевыми ability_modifiers. В deployed — становится container и при наполнении экипажем образует composite unit.

mutable {
  mode:             stowed | deployed | transitioning
  transition_ticks: int             # тиков на смену режима
}
# stowed:   payload активен, ability_modifiers = 0.0, container недоступен
# deployed: payload + container активны
ПримерStowedDeployed
Станковый пулемётПереносится пехотой, modifiers=0Container: расчёт 2 чел., SHOOT↑ MOVE=0
Буксируемая пушкаНа прицепе грузовика, modifiers=0Container: расчёт 4 чел., BLAST↑ MOVE=0
Пункт связиВ кузове, modifiers=0Container: оператор, comms_bonus↑

09 Четыре базовых приказа

Приказ = намерение. Поведение при выполнении определяет ROE, а не приказ.
MOVE
Выдвинуться в точку.
target: Vector3
HOLD
Удерживать точку.
target: Vector3
SUPPORT
Поддержать юнит или зону. Барраж = SUPPORT с целью-зоной.
target: unit_id | Vector3, radius
SPOT
Разведать область без перемещения. Результат → Push-report spot_result (priority 1).
target: Vector3 | unit_id, radius

SPOT — особенности

СвойствоЗначение
ПеремещениеЮнит не двигается к цели
РезультатНемедленный Push-report типа spot_result в net_data
Приоритет отчётаВсегда priority = 1 (критично — за этим специально посылали)
abilities.see → 0SPOT выполним. confidence → 0.0. «Ничего не вижу» — тоже разведывательная информация.
abilities.see < thresholdConfidence низкий, вплоть до статуса stale. Данные не блокируются, но помечаются.
ROE при SPOTЮнит не переходит в ENGAGE (если только Survival не форсирует)

Структура приказа (9-line)

Order {
  L1: type      # MOVE | HOLD | SUPPORT | SPOT
  L2: target    # unit_id или Vector3
  L3: radius    # для SUPPORT / SPOT по зоне
  L4: expire    # секунды | null
  L5: logistics_directive  # опционально — см. ниже
}

10 Logistics directive (L5)

Commander может зафиксировать логистическое решение явно через опциональное поле L5. Без него ROE вычисляет логистические под-действия самостоятельно. С ним — Commander управляет напрямую, ROE не переопределяет (только Survival может прервать).

L5: logistics_directive {
  action:   MOUNT | DISMOUNT | LOAD | UNLOAD | DEPLOY | STOW
  asset_id: asset_id | null   # на каком ассете
  timing:   before | on_arrival | after
}
timingСемантика
beforeСначала выполнить логистику, потом приказ
on_arrivalДобраться до точки, потом выполнить логистику
afterВыполнить приказ, потом логистику

Примеры

# Экипаж покидает машину по прибытии
{ L1: MOVE, L2: pos_X, L5: { action: DISMOUNT, timing: on_arrival } }

# Развернуть орудие здесь прямо сейчас
{ L1: HOLD, L2: current_pos, L5: { action: DEPLOY, asset_id: gun_44, timing: before } }

# Поднять БПЛА, выполнить разведку зоны Q, оставаясь на точке
{ L1: SPOT, L2: zone_Q, L3: 800, L5: { action: MOUNT, asset_id: uav_7, timing: before } }

# Отстреляться и свернуть орудие
{ L1: SUPPORT, L2: target_zone, L5: { action: STOW, asset_id: gun_44, timing: after } }

Авто-режим (без L5)

Под-действиеУсловие автозапуска ROE
MOUNTПеред MOVE если есть assigned контейнер и он готов
DISMOUNTПри FALLBACK если контейнер не может двигаться
DEPLOYВход в зону Red, mutable стоит на позиции (по ROE)
STOWMOVE-приказ, mutable развёрнут и нужно переместиться
LOAD / UNLOADПо asset_policy Commander'а или при пополнении supply

11 Восприятие (Perception)

ИсточникЧто даёт
Себя (self)Позиция, supply, state, abilities, location — всегда доступно
Прямое (SEE)spotted[] в vision_range × abilities.see; terrain рядом
Сеть (net_data)Report'ы юнитов и Commander'а, spot_result от разведки
spotted_object {
  position, type    # unit | composite_unit | asset | unknown
  classification    # vehicle | soldier | structure | ... (перцептивный слой)
  confidence, side, side_confidence
  # Для контейнеров:
  is_container:     bool
  contents_visible: bool     # только при высоком see + малая дистанция
  contents_estimate { crew_present: bool, cargo_estimate: int, confidence: float }
}
Внешний наблюдатель видит контейнер как единый объект. Содержимое — с очень низким confidence. «Видели БМП» ≠ «знаем сколько десанта внутри».
РежимУсловиеИсточник решений
Connectedcomms = onlineПриказы + ROE + net_data
Degradedcomms = degradedПоследний приказ + ROE + частичный net_data
Autonomouscomms = offlineТолько ROE + личное восприятие
Autonomous — инициатива снизу. Именно здесь партизан выдвигается на помощь без приказа.

13 World Data и ZoneMap

World Data — объективная

Один, общий, принадлежит движку (NavigationServer3D). Domain composite unit наследуется от контейнера. При DISMOUNT crew возвращаются к своему domain.

ТипПроходимостьВлияние
openground, airНет модификаторов
roadground, airБонус к move_speed
urbanground, airROE → urban_combat; modifier: move −0.15
fortifiedground, airСнижение входящего урона
waterair, seaНепроходимо для ground

ZoneMap — субъективная

Green

Контролируем или безопасно

Yellow

Неопределённость, возможен контакт

Red

Вражеский контроль или активный бой

Любой contact report           → green → yellow
Подтверждённый, confidence>0.7  → yellow → red
Верификация чистая              → red → yellow → green
spotted[] всегда перекрывает zone_cache. Вижу сам — доверяю глазам, не карте.

14 ROE — Правила применения силы

stance не является частью приказа. ROE — доктрина, наследуется каскадом HQ → Commander → Unit.

Personality

ПараметрДиапазонОписание
risk_tolerance0.0–1.0Готовность действовать при угрозе
loyalty0.0–1.0Реакция на состояние союзников
initiative0.0–1.0Автономия при degraded/offline comms

Drives

can_execute = f(ability_score, path_exists, supply > threshold,
               target_reachable, container_ready)
# container_ready: контейнер исправен и crew >= crew_threshold

perceived    = threat_raw * (1 - risk_tolerance)
social       = priority_delta * loyalty
search_drive = priority_delta * initiative * (1 - best_asset_score)

Дерево решений

0
Нет приказа + offline
initiative > 0.7 and priority_delta > 0.6 → AUTO_ORDER else → HOLD
1
Есть явная logistics_directive (L5)?
выполнить в указанный timing # Только Survival может прервать
2
Можем выполнить приказ?
can_execute > 0.6 and perceived < 0.7 → EXECUTE
3
Почему не можем?
supply < threshold → SEEK_ASSET path blocked → FIND_ALT_PATH ability < threshold → HOLD container_crew < threshold → SEEK_CREW или DISMOUNT + пешком mutable.mode = stowed → DEPLOY (если threat позволяет) search_drive > 0.3 → MOVE(search_area)
4
Угроза перевешивает?
perceived > 0.7 and can_execute > 0.5 → ENGAGE perceived > 0.7 and can_execute < 0.5 → FALLBACK # composite: оценить DISMOUNT vs уйти на транспорте
5
Появилось что-то важнее?
social_delta > current_order.weight → SUPPORT/MOVE
6
Comms offline
initiative * can_execute > 0.5 → автономно (шаги 2–5) else → HOLD

15 Commander

Commander — роль в каждом юните (включая composite). При гибели — следующий по rank принимает командование автоматически.

Уничтожение командиров — легитимная тактика. Система деградирует органически.
CommanderRole {
  is_commander, rank, group_id, baseline_threat
  roe_overrides {
    ability_threshold
    require_mounted: bool     # не отправлять группу пешком
  }
}
next = max(rank) среди unit.state != dead в group_id

# group.units — эффективные юниты (composite там, где образованы)
# Crew внутри composite не видны в верхнем списке
# При dissolve — выходят как отдельные юниты

combat_power — агрегат по abilities

combat_power = mean(
  unit.hp
  * mean(unit.abilities.move, unit.abilities.shoot, unit.abilities.blast)
  * (container.fill_ratio if composite else 1.0)
  for unit in group.effective_units where unit.state != dead
)

Разрешение конфликта разведданных

ШагУсловиеДействие
1 — ВерификацияЕсть свободный юнит с abilities.see > thresholdSPOT(спорная_зона) или MOVE+SPOT, пометить unverified
2 — ВзвешиваниеНет свободного юнитаwinner = max(confidence × qualification)
3 — НеопределённостьДанные критичныplanning_uncertainty = true, худший сценарий
