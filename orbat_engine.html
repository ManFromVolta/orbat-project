01 Концепция

ORBAT — игронезависимый фреймворк управления боем двух и более сторон. Логика боя полностью абстрагирована от движка и ассетов.

Принцип: одна и та же логика должна описывать и общевойсковой бой дивизии NATO, и битву при Хельмовой пади.

ORBAT разрабатывается как самостоятельная стратегическая игра-песочница на Godot Engine. Мир — Оррим, эпоха 1952 н.э.

Граф командования

BattleState
└── HQ
     └── Formation         ← оперативный уровень (GLOBAL)
          └── Commander     ← роль в каждом юните
               └── Unit / CompositeUnit  ← тактический уровень (LOCAL)
                    ├── Order      намерение + опц. директива (L5)
                    ├── ROE        поведение при выполнении
                    ├── Perception что знает юнит
                    └── Assets     inventory / container / mutable

02 Масштабы симуляции

GLOBAL
Formation
Свёрнутая симуляция. Агрегированные abilities, фронтовые линии, логистика. Полная симуляция — по явному флагу.
LOCAL
Unit / CompositeUnit
Полная симуляция в реальном времени. Отдельные юниты, физика, боевые машины с экипажами.
PERSONAL
Перспектива игрока
LOCAL с фокусом внутри конкретного юнита. Отдельного масштаба боевой логики не образует.

Каскад Order → Objective

🌍
GLOBAL
→

↓
переход
Order сверху становится Objective снизу. Рекурсивно на любую глубину.
🗺

→
HQ Formation получает Objective через Net Data → распределяет Orders самостоятельно
Трансляция GLOBAL Order → LOCAL Objective идёт через Net Data. Локальный Commander периодически обращается к net_data для отслеживания изменений Objective. Латентность = реалистичная задержка коммуникации.

Агрегирование LOCAL → GLOBAL

ТриггерОписание
По расписаниюБазовый режим. Период задаётся в конфигурации Formation.
По явному запросуПринудительное обновление в любой момент.
При значимом изменении combat_powerЕсли изменение превышает formation.sync_threshold — немедленное обновление. Порог задаётся per-Formation.

03 Formation

Formation — организационная единица командной иерархии на оперативном уровне. Не является CompositeUnit. Formation — это граф подчинённых юнитов и формирований под единым командованием.

CompositeUnit

Основа — физический контейнер. Тактический уровень. Конкретные юниты в физических слотах. Распадается при потере crew_threshold.

Formation

Основа — организационный номер. Оперативный уровень. Граф подчинённых Formation и юнитов. На GLOBAL физический слой отсутствует.

echelon — дескриптивный тег, не жёсткий тип: squad | platoon | company | battalion | regiment | division | ... Произвольная строка. Позволяет описывать любую армейскую структуру любой эпохи.

Контракт Formation

Formation {
  id                      # организационный, стабильный
  designation             # '151-й гвардейский мотострелковый полк...'
  echelon                 # произвольный дескриптор
  side

  command_graph {
    hq_unit_id            # текущий Commander в LOCAL
                          # = наибольший по иерархии доступный Commander
    officer_list []       # действующий офицерский состав
    sub_formations []     # подчинённые Formation
  }

  aggregate {             # пересчитывается из LOCAL при sync
    strength              # % от штатного состава
    combat_power          # 0.0–1.0
    supply                # 0.0–1.0
    abilities { move, see, shoot, blast }
  }

  asset_registry { vehicles[], weapons[], supplies{} }

  perception_summary {
    known_enemies [], zone_map_slice
    confidence, last_updated
  }

  position, area_of_operations
  objective               # от вышестоящего Formation, через Net Data
  order
  sync_threshold: float   # порог значимого изменения combat_power
}

Агрегирование abilities на GLOBAL

# MOVE: оперативная мобильность, не тактическая скорость
formation.abilities.move = mean(unit.abilities.move) * logistics_factor * terrain_factor

# SEE: разведывательный потенциал AO (доминируют recon юниты)
formation.abilities.see = max_weighted(recon_units.abilities.see)

# SHOOT / BLAST: совокупная огневая мощь
formation.abilities.shoot = weighted_mean(...)
formation.abilities.blast  = weighted_mean(...)

04 Два профиля: физический и логический

Каждая сущность (юнит, ассет) имеет два профиля. Они хранятся и изменяются независимо, но связаны через modifiers стек.

physical_profile
Используется движком на LOCAL / PERSONAL. Конкретные физические характеристики.
move_speed_mps, hp_max,
attack_range_m, damage,
armor_mm, weight_kg...
(перечень уточняется)
logical_profile
Используется ORBAT на всех масштабах. Пересчитывается каждый тик.
base_abilities { move, see,
shoot, blast } (из XML)
+ modifiers[] стек
base_abilities задаются явно в XML-файле. Modifiers стек пересчитывается каждый тик и учитывает физические повреждения, меняющиеся состояния и внешние обстоятельства.

Юнит как физический ассет

# В Godot:
UnitNode (CharacterBody3D)        ← физическое тело в сцене
  └── OrbatAgent (GDScript)       ← логический агент ORBAT
        ├── logical_profile       abilities, modifiers...
        └── ссылка на physical_profile

XML-файлы архетипов

Все юниты и ассеты описываются в отдельных XML-файлах. Godot читает их при загрузке через XMLParser. Поддерживает моддинг: новый файл в директории подхватывается без изменений кода.

/data/units/infantry_rifleman.xml
/data/assets/weapons/rifle_kaverGewehr.xml
/data/assets/vehicles/panzer14.xml

<!-- Структура файла не окончательная, будет уточняться: -->
<unit id="infantry_rifleman">
  <physical>
    <hp_max>100</hp_max>
    <move_speed_mps>1.4</move_speed_mps>
  </physical>
  <logical>
    <base_abilities move="0.7" see="0.6" shoot="0.6" blast="0.0"/>
    <domain>ground</domain>
  </logical>
  <default_inventory>
    <asset ref="rifle_kaverGewehr"/>
  </default_inventory>
</unit>

Смерть юнита и разрушение ассетов

При уничтожении юнита: физический ассет (тело) остаётся в сцене, логический агент деактивируется. Инвентарные ассеты освобождаются и потенциально доступны для подбора.

Ассеты могут быть частично или полностью уничтожены. При частичном уничтожении CompositeUnit (боевой машины) учитываются потенциально доступные ассеты из останков: остаток топлива, часть боекомплекта, запасные части.

Модель распределения выживших ассетов при уничтожении открыта до реализации механики урона на LOCAL уровне.

05 Архитектура юнита

Юнит — листовой узел. Атомарная единица исполнения приказов. Техника — ассеты-контейнеры. Юнитом является экипаж. Принадлежность к типу хранится в profile — для UI, не для логики.

MOVE
Перемещение. Пехота, танки, кавалерия, корабли.
SEE
Обнаружение в радиусе. Разведка, наблюдатель, БПЛА.
SHOOT
Атака прямым огнём (LOS). Стрелки, танки, лучники.
BLAST
Атака зоны без LOS. Артиллерия, миномёты, авиация.

Система abilities: base + modifiers

base_abilities { move, see, shoot, blast }   # из XML, не меняются в runtime
modifiers[] { source, move, see, shoot, blast, expires }
abilities = clamp(base + sum(modifiers), 0.0, 1.0)   # каждый тик

Пример стека MOVE: пехота под огнём в городе

base
0.70из XML
suppressed
−0.50подавляющий огонь
terrain:urban
−0.15застройка
→ abilities.move
0.05ползком

06 Composite Unit

Образуется когда контейнер укомплектован выше crew_threshold. Идентифицируется по контейнеру — смена экипажа не меняет id.

БМП идентифицируется по корпусу машины, не по составу экипажа.
CompositeUnit { id = f(container.id), composite { container_id, crew[], cargo_units[], cargo_assets[] } }

# Образование / растворение:
crew_filled / required >= crew_threshold → form composite
crew_filled / required <  crew_threshold → dissolve → crew как отдельные юниты

combat_power = mean(unit.hp * mean(abilities.move, .shoot, .blast) * fill_ratio
               for unit in group.effective_units where state != dead)

08 Четыре базовых приказа

Приказ = намерение. Поведение определяет ROE. Одинаково применимы на LOCAL и GLOBAL (где юнитом считается Formation).
MOVE
Выдвинуться в точку.
target: Vector3
HOLD
Удерживать точку.
target: Vector3
SUPPORT
Поддержать юнит или зону. Барраж = SUPPORT с зоной.
target: unit_id | Vector3, radius
SPOT
Разведать без перемещения. Результат → spot_result (priority 1).
target: Vector3 | unit_id, radius
Свойство SPOTЗначение
abilities.see → 0Выполним. confidence → 0. «Ничего не вижу» — тоже разведывательная информация.
abilities.see < thresholdВыполним. Confidence низкий, вплоть до stale.
ROEЮнит не переходит в ENGAGE (если только Survival не форсирует)

09 Logistics directive (L5)

Опциональный параметр приказа. Без него ROE вычисляет логистику самостоятельно. С ним — Commander управляет явно, ROE не переопределяет (только Survival прерывает).

L5: { action: MOUNT|DISMOUNT|LOAD|UNLOAD|DEPLOY|STOW, asset_id, timing: before|on_arrival|after }

# Экипаж покидает машину по прибытии:
{ L1: MOVE, L2: pos_X, L5: { action: DISMOUNT, timing: on_arrival } }

# Поднять БПЛА, разведать зону Q, не двигаясь:
{ L1: SPOT, L2: zone_Q, L3: 800, L5: { action: MOUNT, asset_id: uav_7, timing: before } }

11 Система приоритетов

1
Objective
На LOCAL = Order сверху, полученный через Net Data от глобального командования. Не меняется без смены Objective.
2
Order
Текущий приказ от локального Commander'а.
3
Survival
Только когда выполнение физически невозможно. Может прервать L5 директиву.
Survival не перебивает Objective. Юнит не убегает при первом выстреле.

13 ROE — Правила применения силы

ROE работает одинаково на LOCAL и GLOBAL (где «юнитом» считается Formation с агрегированными abilities).

Дерево решений

0
Нет приказа + offline
initiative > 0.7 and priority_delta > 0.6 → AUTO_ORDER else → HOLD
1
Есть явная L5 директива?
выполнить в указанный timing # только Survival прерывает
2
Можем выполнить?
can_execute > 0.6 and perceived < 0.7 → EXECUTE
3
Почему не можем?
supply < threshold → SEEK_ASSET path blocked → FIND_ALT_PATH ability < threshold → HOLD container_crew < threshold → SEEK_CREW / DISMOUNT mutable.mode = stowed → DEPLOY search_drive > 0.3 → MOVE(search_area)
4
Угроза перевешивает?
perceived > 0.7 and can_execute > 0.5 → ENGAGE perceived > 0.7 and can_execute < 0.5 → FALLBACK # composite: DISMOUNT vs транспорт
5
Что-то важнее?
social_delta > order.weight → SUPPORT/MOVE
6
Comms offline
initiative * can_execute > 0.5 → автономно (шаги 2–5) else → HOLD

14 Commander

Роль в каждом юните. При гибели — следующий по rank принимает командование. На LOCAL роль HQ Formation = наибольший по иерархии доступный Commander.

Уничтожение командиров — легитимная тактика. Система деградирует органически.
CommanderRole { is_commander, rank, group_id, baseline_threat,
  roe_overrides { ability_threshold, require_mounted: bool } }
next = max(rank) среди unit.state != dead в group_id

15 Net Data

qualification = base_skill * unit.hp * abilities.see * (1-threat_level) * logistics.supply
